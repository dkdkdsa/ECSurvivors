
# ECSurvivors
ECS 기반 뱀서라이크 대규모 오브젝트 처리 실험 프로젝트

## 목적  
본 프로젝트는 ECS(Entity Component System)를 활용하여  
뱀서라이크 장르에서 발생하는 대량의 적 및 탄환 처리 성능을 실험하고,  

기존 GameObject + Jobs 기반 구현과 비교하여  
각 아키텍처의 성능 특성과 한계를 분석하는 것을 목표로 합니다.

## TL;DR
사용 엔진 : Unity 6000.3.13f1
핵심 구현 : 공간 해싱 방식을 이용한 커스텀 콜라이더 시스템 / ECB를 사용하여 ECS/UI 통신
검증 : GameObject로 최대한 동일한 방식으로 구현 후 벤치마킹
결과 : 
ECS가 대규모 엔티티 처리에서 GameObject 대비 확실한 성능 우위를 보임
- 10,000 엔티티에서 ECS (0.89ms) / GameObject (2.65ms) (약 3배)
- 50,000 엔티티에서 ECS (9.6ms) / GameObject (627ms) (약 65배)

## 구현 내용
- 충돌 시스템 : Unity의 물리엔진에 의존하지 않고 공간 해싱 방식을 사용하여 직접 구현한 충돌 시스템을 사용하여 충돌 로직이 Job으로 병렬 처리될 수 있게 함
- ECS - Mono 브릿지 패턴 : ECS 쿼리 결과를 UnityEvent로 전달하여 Mono로 구현된 시스템과 쉽게 연동되도록 함
- DynamicBuffer 기반 이벤트 시스템 : 충돌 이벤트/데미지 이벤트 등을 버퍼로 전달하여 생성/사용을 분리
- 레벨업/강화 시스템 : ECB를 통하여 UI/ECS UI에서 선택한 강화 정보를 ECS로 유연하게 전달

## 벤치마크
단순히 ECS가 빠르다는 결론을 내기보다  
공정한 비교를 위해 동일한 로직과 구조를 유지하는 것을 목표로 함

GameObject 구현에서도  
- 공간 해싱 기반 동일 충돌 로직 적용  
- Object Pooling 사용  
- JobSystem을 활용한 병렬 처리  

를 적용하여 최대한 동일한 조건에서 성능을 비교

벤치마크 테스트 환경은 다음과 같음

PC
- CPU : AMD Ryzen 7 9700X
- GPU : Nvidia RTX 5070 12GB
- RAM : DDR5 32GB
- OS : Windows 11 home

환경
- 유니티 버전 : Unity 6000.3.13f1
- 빌드 : IL2CPP Release Build
- VSync : off
- 각 케이스별 3회 측정 후 평균

벤치마크 결과
| Object Count | ECS (ms) | GO (ms) | Ratio |
|---|---|---|---|
| 100 | 0.46 | 0.55 | 1.20x |
| 500 | 0.53 | 0.78 | 1.48x |
| 1,000 | 0.60 | 1.14 | 1.89x |
| 5,000 | 0.69 | 1.60 | 2.33x |
| 10,000 | 0.89 | 2.65 | 2.97x |
| 50,000 | 9.62 | 627.28 | 65.18x |

해석
- ~ 500 엔티티 : 부하가 낮은 환경에서는 아키텍쳐 차이가 크지 않음
- 1,000 ~ 10,00 엔티티 : 2~3배로 유의미한 차이가 벌어지기 시작
- 50,000 엔티티 :   
GameObject는 객체 관리 및 메모리 접근 비용 증가로 인해 급격한 성능 저하 발생  
ECS는 데이터 지향 구조로 인해 캐시 효율이 유지되며 비교적 선형적인 성능 증가를 보임

## 제작 기간
2026.04.16 ~ 2026.04.19 (4일)

## 프로젝트를 통하여 느낀점
- ECS가 병렬 데이터 처리에 뛰어나다는걸 다시 한번 실감함
- 실제 뱀서라이크 장르에서는 10,000+ 이상의 엔티티가 항상 요구되지는 않기 때문에  
  개발 편의성과 생산성을 고려하면 GameObject 기반 접근도 충분히 실용적일 수 있음  
- 다만, 확장성이나 극단적인 상황 대응에서는 ECS가 명확한 강점을 보임
- 향후 모바일 환경에서도 ECS가 우위를 보이는지 보인다면 얼마나 보이는지를 테스트 해보고 싶음
